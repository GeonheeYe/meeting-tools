# 회의 STT 정확도 개선 설계

**날짜**: 2026-03-19
**상태**: 승인됨

## 목표

현재 회의 STT 파이프라인의 주요 문제는 두 가지다.

1. 도메인 용어 오인식
2. 문장 누락 및 잘림

정확도 향상이 최우선이며, 처리 시간이 다소 늘어나는 것은 허용한다. 또한 회의마다 참고 문서를 함께 주는 운영 방식을 전제로 한다.

## 문제 정리

기존 파이프라인은 다음 특성이 있다.

- VAD로 무음 제거 후 Whisper 실행
- 참고 문서에서 추출한 키워드를 `initial_prompt`에 주입
- 화자 수 미지정 시 화자 분리 생략

이 구조는 처리 시간 측면에서는 합리적이지만, 긴 회의를 한 번에 STT하는 방식이라 다음 문제가 남아 있다.

- 긴 파일 후반부에서 문맥 유지가 약해질 수 있음
- 발화 경계 근처 문장이 잘리거나 누락될 수 있음
- 문서 기반 힌트가 단순 키워드 나열 수준이라 용어 보정력이 제한적임

## 선택한 접근

다음 세 가지 접근 중 `청크 기반 STT + 문서 강화`를 채택한다.

1. 문서 강화 + 후처리 교정 중심
2. 청크 기반 STT + 문서 강화
3. 2-pass STT

선택 이유:

- `문장 누락/잘림`은 후처리만으로 해결하기 어려움
- `도메인 용어 오인식`은 문서 기반 힌트와 교정으로 개선 가능
- `2-pass STT`는 잠재력은 크지만 현재 코드베이스 기준으로는 복잡도와 비용이 과함

## 설계 개요

파이프라인의 핵심 역할을 다음과 같이 재정의한다.

- `context_loader`: 문서에서 키워드뿐 아니라 우선 용어 구조를 추출
- `VAD`: 무음 제거기가 아니라 청크 경계 탐지기 역할
- `chunker`: 긴 회의를 안정적으로 나누는 전용 단계
- `transcribe`: 청크별 Whisper 실행
- `merge`: 청크 결과 병합 및 중복 제거
- `post-correction`: 문서 근거 기반의 보수적 용어 교정

## 파이프라인 흐름

```text
/meeting audio.wav 제목 [N명] [키워드] [참고문서들]

1. context_loader
   - 핵심 용어 추출
   - 약어/정식명칭 매핑 추출
   - 우선 용어 리스트 생성

2. VAD 기반 경계 탐지
   - 3초 이상 침묵을 청크 경계 후보로 사용

3. chunker
   - 최대 길이 기준으로 추가 분할
   - 경계 보호를 위한 오버랩 추가

4. Whisper
   - 청크별 STT 실행
   - 문서 기반 프롬프트 주입

5. merge
   - 시간순 병합
   - 오버랩 구간 중복 제거

6. post-correction
   - 문서에 실제 존재하는 용어만 교정

7. summarize / Notion
   - 기존 흐름 유지
```

## 청크 전략

청크 기준은 다음으로 고정한다.

- 기본 경계: `3초 이상 침묵`
- 최대 청크 길이: `90초~120초`
- 최소 청크 길이: `15초`
- 청크 오버랩: `0.5초~1초`

세부 규칙:

- `3초 이상 침묵`이 있으면 우선 경계 후보로 채택한다.
- 침묵 경계가 없더라도 청크가 너무 길어지면 최대 길이 기준으로 분할한다.
- 너무 짧은 청크는 앞뒤 청크와 병합한다.
- 청크 경계에서 문장이 잘리지 않도록 앞뒤 오버랩을 둔다.
- 병합 시 오버랩 구간의 중복 문장은 텍스트 유사도 기준으로 제거한다.

이 전략의 목적은 무음을 많이 삭제하는 것이 아니라, 긴 파일을 안정적으로 분해해 Whisper가 문맥을 놓치지 않게 만드는 것이다.

## 문서 기반 정확도 향상 전략

### 1. 프롬프트 구조화

`initial_prompt`를 단순 키워드 나열이 아니라 아래 구조로 구성한다.

- 회의 주제
- 제품명 / 프로젝트명
- 기능명 / 과제명
- 약어와 정식명칭

예시:

```text
회의 주제: 목표합의서
핵심 용어: AEGIS-AP, WiNG, VOC 기반 AI 분석 기능, 품질 분석 AI 기능
표기 참고: VOC, VoC, Domain Agent
```

### 2. 용어 우선순위

문서에서 추출한 용어에 우선순위를 둔다.

1. 제품명 / 프로젝트명
2. 기능명 / 과제명
3. 일반 명사

이 우선순위는 프롬프트 구성과 후처리 교정에 함께 사용한다.

### 3. 보수적 후처리 교정

후처리 교정은 다음 원칙을 따른다.

- 문서에 실제 존재하는 표현만 교정
- 약어와 정식명칭의 표기 흔들림은 정규화
- 문서에 없는 단어를 추측해서 삽입하지 않음

즉, 교정은 생성이 아니라 정합성 보정 역할만 맡긴다.

## 컴포넌트별 변경 방향

### `context_loader.py`

- 현재 키워드 추출 외에 `용어 사전` 구조를 반환하도록 확장
- 반환 예시:
  - `key_terms`
  - `canonical_terms`
  - `alias_map`
  - `doc_content`

### `pipeline.py`

- 전체 파일을 바로 STT하지 않고 청크 기반 실행으로 전환
- VAD는 무음 제거가 아니라 경계 계산에 사용
- 결과 JSON에 청크 정보와 적용 전략 메타데이터를 저장

예시:

```json
{
  "title": "...",
  "date": "...",
  "speaker_count": 1,
  "transcript": "...",
  "doc_content": "...",
  "vad_applied": true,
  "chunking": {
    "silence_ms": 3000,
    "max_chunk_sec": 120,
    "overlap_sec": 1.0
  }
}
```

### `transcribe.py`

- 청크별 Whisper 호출 지원
- 청크별 프롬프트 주입 지원
- 병합 가능한 세그먼트 메타데이터 유지

### 병합 단계

- 시간축 기준 정렬
- 오버랩 영역 중복 제거
- 동일 문장 반복 제거
- 청크 경계에서 잘린 문장을 가능한 범위에서 자연스럽게 결합

## 오류 처리 원칙

- 문서가 없어도 기본 STT는 동작해야 함
- 문서 파싱 실패 시 전체 파이프라인이 죽지 않고 경고 후 진행
- 청크 하나가 실패해도 전체 실패 대신 재시도 또는 부분 실패 기록을 남김
- 병합 실패 시 원본 청크 결과를 보존해 디버깅 가능하게 함

## 검증 기준

성능 개선 여부는 다음 기준으로 판단한다.

1. 같은 회의 파일에서 환각 문장 수 감소
2. 문서 기반 핵심 용어 매칭률 증가
3. 사람이 읽을 때 문장 단절 감소
4. 기존 단일 STT 또는 기존 VAD 결과보다 중반 이후 문맥 보존 개선

## 비교 대상

같은 입력 파일에 대해 최소 3개 결과를 비교 가능해야 한다.

1. 원본 직처리
2. 기존 VAD 기반 단일 STT
3. 청크 기반 VAD + 문서 강화 STT

## 제외 항목

- 실시간 STT
- 스트리밍 UI
- 다단계 재인식 기반의 2-pass STT
- 문서에 없는 표현을 생성적으로 보정하는 공격적 후처리

## 구현 우선순위

1. 청크 경계 탐지와 청크 실행 구조 추가
2. 문서 기반 프롬프트 구조화
3. 병합 및 중복 제거
4. 문서 근거 기반 용어 교정
5. 비교 메타데이터 및 검증 도구 추가
